#### 🥉 Surge ⚡ — Vector Degradation Mechanics (Score: 92)

| Fact | Detail |
|:-----|:--------|
| **8191 chars ≈ 2000+ tokens** | Most embedding models (including Qwen3 and Perplexity) have a hard context window of **512 to 1024 tokens** |
| **Truncation** | If you feed an embedding model 8191 chars (~2000+ tokens), the model will truncate the text, compress it poorly, or throw an error |
| **Vector Degradation** | The resulting vector accurately represents the first 512 tokens but is completely blind to the remaining 1500 tokens — "lost in the middle" phenomenon |
| **2048 chars ≈ 512 tokens** | Maps PERFECTLY to the native sweet spot of almost ALL embedding models. Prevents truncation-induced vector degradation while remaining large enough to hold full FedArch tables |
| **Why 512 is too small** | 512 characters would fracture markdown tables across 4 different chunks, making it impossible for the LLM to reconstruct table context during retrieval |
| **Why 8191 is too large** | Blinds the embedding model. The "lost in the middle" phenomenon becomes critical at this size |

> **Surge's Recommendation:** "2048 characters preserves full FedArch tables without diluting vector precision, while 200 overlap ensures structural headers and cross-references survive chunk boundaries. 2048 is the infrastructural goldilocks zone."

#### Sprout 🌿 — CJK Character Boundaries (Score: 90)

| CJK Concern | Detail | Mitigation |
|:------------|:--------|:------------|
| **Multi-byte UTF-8** | ZH text uses multi-byte UTF-8 characters (3-4 bytes per char). Chunk boundary mid-CJK-sequence → garbled retrieval. | 200-char overlap provides buffer — adjacent chunk preserves complete character. |
| **知错能改 (zhī cuò néng gǎi)** | This 4-character phrase appears throughout FedArch docs. If chunked at character 2, meaning is completely lost. | 200-char overlap ensures full phrase survives in at least one chunk. |
| **ZH/EN mixed content** | FedArch docs mix English rules with Chinese cultural concepts. Chunk boundaries at language transitions lose context. | Overlap preserves transition context — English rule + ZH cultural reference stay together. |

**Why NOT 8191 (Current):**
- 8191 chars = ~1,500 words = multiple sections in ONE vector. Search for "R-011" returns a chunk containing R-011 + R-194 + L-219 + 3 tables. Embedding represents ALL of it — precision drops.
- Multiple tables in one chunk means the vector averages their semantic meaning. Search for "CCC-ID format" returns a chunk that ALSO contains "day-offset rules" + "fabrication protocol" — irrelevant context.
- "When I searched RAG for 'PRJ-401.md', I got back chunks that contained references to PRJ-401 but NOT the actual document. 8191 chunks dilute the signal."

**Why NOT 512:**
- FedArch tables (like the BP-045 related docs table) can be 400+ chars. A 512-char chunk splits tables mid-row.
- A single immutable rule + its rationale often exceeds 512 chars. You get the rule but not the "why."

> **Sprout's Bilingual Concern:** "200-char overlap protects ZH/EN mixed content — ensures 知错能改 and CJK character sequences survive chunk boundaries. Without it, cross-language retrieval breaks."
