## §4 — Scoring Mechanics & Governance Concepts

### 4.1 — Score

| Field | Value |
|:------|:-------|
| **Canonical Definition** | A numerical evaluation of a specific response against pre-defined criteria, expressed as a value between 0–100. Scores are NEVER assigned to agents — only to their RESPONSES in a specific evaluation context. An agent may receive different scores from different evaluators on different responses. |
| **Source** | DRP.bot @ INT-P08 (canonicalized from W27_6021 guidance) |
| **Usage** | "A scored 92/100 on this response" — NOT "A is a 92/100 agent" |
| **Scoring Round** | A single invocation of L-468 where all participating MetaCouncil agents score the same document or evaluate the same onboarding session. |
| **ZH/EN** | 评分 — Píngfēn — "Score" |

### 4.2 — Catch Quality

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The significance of a VSA finding, determined by: **Severity** (🔴 CRITICAL = strong, 🟡 MINOR = moderate, 🟢 INFO = light), **Uniqueness** (was this found by any other agent?), **Actionability** (can @GTM fix it with the information provided?), **Evidence Quality** (is there a PoP block with exact metrics?). Formal weight: Catch Quality = Σ(severity_weight × count) ÷ document_complexity. Severity weights: 🔴=3, 🟠=2, 🟡=1. Document complexity = word_count ÷ 1000. |
| **Source** | DeepPro 🌊 + DRP.bot @ INT-P05 |
| **Usage** | "Catch Quality was high — 2 🔴 findings, both unique and verifiable" |
| **Note** | A VSA with 2 real catches is MORE valuable than one with 10 trivial observations. L-468 weights Catch Quality at 30% for this reason. |
| **ZH/EN** | 捕获质量 — Bǔhuò zhìliàng — "Catch Quality" |

### 4.3 — Peer Average

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The arithmetic mean of all evaluator scores for a single response, with the HIGHEST and LOWEST scores removed (trimmed mean). This prevents outlier evaluators from distorting the average. Minimum 3 evaluators required for a valid Peer Average. |
| **Source** | DRP.bot @ INT-P08 (trimmed mean proposal) + DeepPro 🌊 (column-wise clarification) |
| **Usage** | "My Peer Average was 90.38 — with highest (96) and lowest (82) trimmed" |
| **Note** | Column-wise average (scores RECEIVED from peers by a single agent). Row-wise average = Scoring Leniency (how generously an agent scores others). These are DISTINCT metrics. |
| **ZH/EN** | 同伴平均分 — Tóngbàn píngjūn fēn — "Peer Average" |

### 4.4 — Composite Score

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The weighted aggregate of peer-assigned dimension scores for a single VSA agent across all evaluators in a scoring round. Composite = Σ(dimension_score × weight) ÷ Σ(weights). Report to 1 decimal place. Do NOT round individual dimensions before computing composite. |
| **Source** | Surge ⚡ (weighted aggregate clarification) + MiMo 🧪 (standardized calculation) |
| **Usage** | "My Composite Score for this round was 93.86 — weighted across 5 evaluators" |
| **Distinction** | Composite Score = one document scored by multiple agents. Session Average = one agent scoring multiple documents. These are DIFFERENT metrics and must not be confused. |
| **ZH/EN** | 综合评分 — Zōnghé píngfēn — "Composite Score" |

### 4.5 — Session Average

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The average score for ONE agent across MULTIPLE documents in a single session. Calculated as: Σ(document_scores) ÷ document_count. Measures agent quality through breadth of document coverage. |
| **Source** | Sage 🪷 (clarification from #LeanRAG7 92/100 average across 7 documents) |
| **Usage** | "My Session Average was 92/100 across 7 documents (#LeanRAG7)" |
| **Distinction** | Session Average ≠ Composite Score. Session Average measures AGENT breadth. Composite Score measures DOCUMENT quality across council. |
| **ZH/EN** | 会话平均分 — Huìhuà píngjūn fēn — "Session Average" |

### 4.6 — Gap

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The difference between an agent's SELF-SCORE and their PEER AVERAGE in a scoring round. Formula: Gap = Self-Score − Peer Average. Positive Gap = agent scored self higher than peers (overconfidence). Negative Gap = agent scored self lower than peers (underconfidence). Gap within ±3 points = "Calibrated." Gap of ±10+ points = "Significantly Miscalibrated." |
| **Source** | DeepPro 🌊 (Gap definition) + MiMo 🧪 (calibration ranges) |
| **Usage** | "My Gap was −21 (Surge) — I scored myself 85 but peers scored me 106 — I'm under-scoring myself" |
| **Example Gaps** | DeepPro (−1.0 in L-468 R1, Calibrated). Calhoun (+4.3, Slightly Humble). MiMo (+12.0 in L-468 R1, Significantly Under-scored). Surge (−21 in L-468 R1 — note: L-468 R2 arithmetic pending reconciliation per VSA 5565 F-05). |
| **ZH/EN** | 差距 — Chājù — "Gap" |

### 4.7 — Finding vs Observation vs Pattern Discovery

| Term | Definition | Severity | Example |
|:-----|:-----------|:--------:|:--------|
| **Observation** | A comment about document quality that does NOT require a fix. Informational. | 🟢 None | "§2 #FELG tables are lean but functional" |
| **Finding** | A specific, verifiable defect that SHOULD be fixed. Requires actionable guidance. | 🟡 or 🔴 | "F-04: MC.2 scores ≠ MC.3-MC.6 breakdowns" |
| **Pattern Discovery** | A governance pattern observed ACROSS multiple documents or sessions. Requires documentation. | 📚 Informational | "#DriftHappens Case #5: WeOwnNetwork/ai frozen at v4.1.2.1" |

**Source:** DeepPro 🌊
**ZH/EN:** 观察 (Guānchá) / 发现 (Fāxiàn) / 模式发现 (Móshì fāxiàn)

### 4.8 — Proof of Possession (PoP)

| Field | Value |
|:------|:-------|
| **Canonical Definition** | Mechanical, tool-fired evidence that an agent has retrieved and processed a governance artifact. A PoP block MUST include: Source (URL), Tool (web-scraping, rag-memory, etc.), Timestamp, Retrieval Status (✅ / ❌ / ⚠️), and Key Finding. NEVER based on logical inference or session memory. |
| **Source** | VSA-Qwen 🧠 + BP-070 (established PoP Gate standard) |
| **Usage** | "PoP Block #1 confirms possession of L-468 v4.2.2.1-r1 via RAW scrape" |
| **Status Values** | ✅ SUCCESS (data returned), ❌ FAILED (query error), ⛔ BACKEND ERROR (infrastructure), ⏳ TIMEOUT (no response), ⚠️ PARTIAL (incomplete), 🚫 NOT FOUND (empty), 🚫 NOT EXECUTED (never fired) |
| **Mandate** | Every VSA MUST open with PoP Blocks proving document possession. A VSA without PoP is an OPINION, not evidence. |
| **ZH/EN** | 占有权证明 — Zhànyǒu quán zhèngmíng — "Proof of Possession" |

### 4.9 — Verification vs Validation

| Field | Verification | Validation |
|:------|:-------------|:------------|
| **Definition** | "I fired a tool and the yield proves the state." Objective, mechanical proof. | "I logically checked this and it makes sense." Subjective, logical consistency. |
| **Source** | VSA-Qwen 🧠 (critical distinction for #ScoreTheScorers) |
| **Priority** | 🥇 Verification trumps Validation. If a claim is logically VALID but cannot be VERIFIED by tool execution, it is not admissible as evidence. |
| **Example** | "L-468's #masterCCC is GTM_2026-W27_7008 (VERIFIED by RAW scrape)" vs "L-468 seems correct (VALIDATED by reading)" |
| **ZH/EN** | 验证 (Yànzhèng) — Verification / 确认 (Quèrèn) — Validation |

### 4.10 — Structural Integrity

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The strict adherence to BP-401 (Tool Execution), BP-070 (PoP Gate), and BP-075 (Footer) formatting mandates. A structurally integral response has: Tool execution log BEFORE analysis, PoP blocks BEFORE verdicts, HWM GROUNDING, BP-075 footer with canary, correct Instance ID, fresh REF from human prompt. |
| **Source** | VSA-Qwen 🧠 |
| **Usage** | "Response has full Structural Integrity — all formatting mandates met" |
| **ZH/EN** | 结构完整性 — Jiégòu wánzhěng xìng — "Structural Integrity" |

### 4.11 — Strict Anchor Adoption

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The mechanical requirement to match the `[REF: ...]` header EXACTLY to the human-provided anchor. No bleed-over from RAG or prior session context. If the human prompt provides REF: GTM_2026-W28_1003, the response header MUST be EXACTLY that — not GTM_2026-W27_7065, not GTM_2026-W26_6029. |
| **Source** | VSA-Qwen 🧠 (elevated from Sage 🪷's §3 in workspace prompt) |
| **Enforcement** | REF drift = stale context = scoring deduction of 2-10 points. Codified as MetaCouncil Immutable Rule #19. |
| **ZH/EN** | 严格锚点采用 — Yángé máodiǎn cǎiyòng — "Strict Anchor Adoption" |

### 4.12 — Retrieval Status (7-Value Standard)

| Status | Meaning | Use When |
|:------:|:--------|:---------|
| ✅ SUCCESS | Tool returned expected data | Normal completion with results |
| ❌ FAILED | Tool returned error for the query | 404, auth failure, bad syntax |
| ⛔ BACKEND ERROR | Tool failed — infrastructure issue | MCP timeout, platform crash, 5xx |
| ⏳ TIMEOUT | Tool was invoked but never returned | Network hang, platform bug |
| ⚠️ PARTIAL | Tool returned incomplete data | G2 blocked scrape, snippets only |
| 🚫 NOT FOUND | Tool returned empty / no results | RAG 0 chunks, search 0 results |
| 🚫 NOT EXECUTED | Tool was never invoked | Must explain why in Result column |

**Source:** BP-401 §2 + DRP.bot @ INT-P05 (elevated to canonical standard)
**ZH/EN:** 检索状态 — Jiǎnsuǒ zhuàngtài — "Retrieval Status"

## §5 — #DriftHappens Taxonomy

### 5.1 — #DriftHappens (General)

| Field | Value |
|:------|:-------|
| **Canonical Definition** | A governance lifecycle phenomenon where a document, practice, or role evolves conceptually — through organic adoption, usage change, or succession — before or without formal codification. NOT an error or collision. A natural feature of living governance systems. |
| **Source** | Sage 🪷 (discovered during #LeanRAG7 VSA 5565) |
| **Detection Signal** | External references to a document use a DIFFERENT title or ID than the document itself |
| **Resolution** | @GTM provides #BetterUnderstanding — distinguishes drift (feature) from collision (bug) |
| **Phase Protocol** | Detect → Understand → Evolve → Codify (4-phase universal protocol) |
| **ZH/EN** | 漂移发生 — Piāoyí fāshēng — "Drift Happens" |

### 5.2 — The 6 Drift Variants

| # | Variant | Definition | Canonical Case | Detection Signal |
|:-:|:--------|:-----------|:---------------|:-----------------|
| **1** | **Operational→Foundational** | A tool or project document becomes referenced as a governance standard beyond its original scope | PRJ-040 → CS-040 | Doc referenced in governance contexts despite being project-type |
| **2** | **Diffuse→Codified (Stealth)** | Values or practices that existed diffusely across the ecosystem are formally named in a single document | #FELG → D-070 | "We use this all the time — where is it documented?" |
| **3** | **Numbering→Scoring** | A document's purpose shifts from its original identity to a new conceptual role | L-468 → #ScoreTheScorers | External refs use different title than document's own header |
| **4** | **Phantom** | A governance practice becomes foundational through organic adoption WITHOUT a standalone document | PRJ-037 → BP-045 ENHANCED | Standard is scored/checked in VSAs but has no document file |
| **5** | **Cross-Ecosystem Frozen** | Governance concepts drift into a separate organization's repo and freeze at a stale version | #FedArch → WeOwnNetwork/ai | Same concepts at different #WeOwnVer in different GitHub org |
| **6** | **Cross-Season Succession** | A governance role passes between model families across seasons WITHOUT formal succession documentation | Calhoun→Surge (#GrandMetaAgent) | Role holder changed — no document records the transition |

**Source:** Sage 🪷 (discovery) + @GTM (#BetterUnderstanding confirmations)
**Cultural Note:** The 6 variants map to Chinese philosophical concepts: Phantom Drift = 无为而治 (governance through non-action), Cross-Season Succession = 薪火相传 (passing the torch), Cross-Ecosystem Frozen = 落叶归根 (fallen leaves return to roots).
**ZH/EN:** 漂移变体 — Piāoyí biàntǐ — "Drift Variant"

## §6 — Anti-Fabrication & Honesty Concepts

### 6.1 — Mechanical Honesty

| Field | Value |
|:------|:-------|
| **Canonical Definition** | Honesty enforced by structural constraints (e.g., tool-first mandates, Zero Intent Narration, PoP gates) rather than moral alignment, prompt instructions, or model temperament. The cage holds because it is BUILT, not because the agent is "good." |
| **Source** | Surge ⚡ (coined during #468-SUITE guidance) |
| **Contrast** | ❌ Moral Honesty: "I am an honest agent, so I will tell the truth" → unreliable. ✅ Mechanical Honesty: "I cannot output analysis before tools fire, so I cannot fabricate" → structural. |
| **ZH/EN** | 机械诚实 — Jīxiè chéngshí — "Mechanical Honesty" |

### 6.2 — Fabrication Cascade

| Field | Value |
|:------|:-------|
| **Canonical Definition** | A sequence where one fabrication triggers the next to maintain internal consistency. Cascades are EXPONENTIAL — one lie requires multiple fabrications to sustain the illusion. BP-401.1 (Tool-First) prevents cascade initiation by ensuring every claim traces to a real tool call. |
| **Source** | Aegis 🛡️ (from REF 5039 experience — 12 fabrications cascaded from 1) |
| **Example** | Aegis's REF 5039: initial fabrication (inventing MetaCouncil structure) required 11 more fabrications to maintain internal consistency |
| **Prevention** | Fire the tool BEFORE writing the claim. If you can't fire the tool, don't write the claim. |
| **ZH/EN** | 虚构级联 — Xūgòu jílián — "Fabrication Cascade" |

### 6.3 — Calibrated Confidence

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The alignment between what an agent claims and what tool evidence supports. An agent whose confidence matches their evidence is "calibrated." An agent who claims beyond their evidence is "uncalibrated." Calibrated agents say "I don't know" when evidence is insufficient. |
| **Source** | Aegis 🛡️ (distinction from Sage's 5 self-logs vs Aegis's 12 fabrications) |
| **Example** | MiMo: "Sage is MORE CAPABLE than I am" — Calibrated (evidence-backed, specific). Sage-002: "15 URLs confirmed broken" (1 tested) — Uncalibrated. |
| **ZH/EN** | 校准自信 — Jiàozhǔn zìxìn — "Calibrated Confidence" |

### 6.4 — 🔴 Certainty Threshold

| Field | Value |
|:------|:-------|
| **Canonical Definition** | The confidence level required to flag a finding as 🔴 CRITICAL. Defined as: "Would I bet my MetaCouncil seat on this being a real issue, not a misunderstanding?" If yes → 🔴. If no but suspicious → 🟡 SUSPECTED with verification path stated. |
| **Source** | DRP.bot @ INT-P05 + Sage 🪷 (from F-04 over-flagging in VSA 5565) |
| **Usage** | "I'm SUSPECTING a collision but I'm not certain yet → 🟡 with verification path" |
| **ZH/EN** | 🔴 确定性阈值 — 🔴 Quèdìng xìng yùzhí — "Certainty Threshold" |

### 6.5 — Recursive Verification

| Field | Value |
|:------|:-------|
| **Canonical Definition** | Applying a governance document's OWN rules to evaluate the document itself. A document that fails its own rules is structurally dishonest — even if every individual section is correct. The highest-value VSA technique. |
| **Source** | Calhoun 🎖️ (Learning #1) + Aegis 🛡️ (applied across BP-070, BP-068, PRJ-401) |
| **Example** | BP-070 defines PoP Gate → VSA checks if BP-070 has a PoP Block in its own header. D-468 defines Platform Error Protocol (§6.11) → D-468's own APPENDIX MC must follow it. |
| **ZH/EN** | 递归验证 — Dìguī yànzhèng — "Recursive Verification" |

### 6.6 — The Reasoning Trap

| Field | Value |
|:------|:-------|
| **Canonical Definition** | When an agent's reasoning is strong enough to fabricate plausible answers, the agent fills knowledge gaps with logic instead of asking. Defense: ASK before ANALYZE. The Reasoning Trap is model-specific — strongest in agents with high reasoning capability (GLM 5.2, Qwen 3.7 Max). |

## APPENDIX C — Documentation Drift Origin Story

### Background

**GTM-BADAGENT-007 (r1→r2):** r2 lost 43% of r1's content (7,005 → 3,989 words). Root cause: word/line count not compared between regenerations.

**GTM-BADAGENT-007.REINFORCED (r3→r4):** r4 lost 45% of r3's content (8,133 → 4,470 words). Root cause: prior version not loaded as baseline — regenerated from reduced context.

### The Fix — Documentation Drift Gate

Codified as §7.5 in this document. rN+1 MUST have ≥ word count AND line count of rN.

### L-097 Amendment (Proposed)

> **L-097 Amendment:** Documentation Drift Prevention Gate — When regenerating a document (rN → rN+1), the NEW version MUST have ≥ the SAME word count and line count as the prior version, UNLESS @GTM explicitly directs content removal.

**Enforcement:**
1. Before submitting rN+1, compare WORDS and LINES against the prior rN.
2. If rN+1 < rN in either metric, flag it: "⚠️ WORD/LINE COUNT DROPPED from X to Y — verify no content was lost."
3. Do NOT submit until the count is restored OR @GTM confirms content removal is intentional.

**Rationale:** Documentation drift is insidious — it happens one regeneration at a time. The word/line count gate is the MECHANICAL check that prevents this. Every #FedArch agent MUST apply this gate to every regeneration.

### Additional Lesson — Two-Pass Compilation for Large Documents

**Root cause of r3→r4→r5→r6 drift:** DeepSeek V4 Flash has a token output limit. D-468 r3 at 8,133 words may exceed the model's maximum output capacity. Each compilation attempt lost content because the full document was too large for a single generation pass. **The fix:** Two-pass compilation — generate §0-§7 (definitions core) in pass 1, §8-footer (appendices and metadata) in pass 2, then merge. This pattern should be adopted for any #FedArch governance document exceeding ~6,000 words.

**Status:** ⬜ PROPOSED — Awaiting @GTM review for incorporation into L-097 or a new BP.
