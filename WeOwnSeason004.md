## 1. 🔍 #StartWithWhy

### Why This Document Exists

> **Because guessing your version number is the #1 preventable error in FedArch.**

Time and again, agents draft documents with wrong version numbers. `v0.1.0.0`. `v1.0`. `v4.1.4.1-r0`. The wrong week offset. The wrong season. Every single one of these errors was preventable if the agent had checked a simple, up-to-date, authoritative reference before writing.

**This document is that reference.**

| "What week are we in?" | **§4 — Current Week Quick Reference** — glance and go |
| "What version do I use TODAY?" | **§5 — Day-by-Day Reference** — every day of W26 mapped |
| "What season are we in?" | **§6 — Season Overview** — full season at a glance |

## 9. 🚨 COMMON ERRORS & HOW TO AVOID THEM

| # | Error | Example | Root Cause | **How to Avoid** |
|:-:|:------|:--------|:-----------|:-----------------|
| **1** | **Pre-release versioning** | `v0.1.0.0` | Old software development habits | **Scrape this doc. Read §4. Use `v4.1.4.1-r1`. There is NO v0.x in #FedArch.** |
| **2** | **r0 instead of r1** | `v4.1.4.1-r0` | Assuming zero-indexed revisions | **There is NO r0. r1 is the starting point. ALWAYS.** |
| **3** | **Wrong week offset** | `v4.1.1.1` for W26 | Guessing the week number | **W26 = Week 4 of S004 = `v4.1.4.1`. Use the formula: W(N) → Week N-22.** |
| **4** | **Using calendar week as version** | `v4.1.26.1` | Confusing W26 (calendar) with Week 26 (doesn't exist) | **W26 = Week 4 of the season. The version is `v4.1.4.1`, not `v4.1.26.1`.** |
| **5** | **No version at all** | Missing #WeOwnVer header | Forgetting to add version | **ALL governance documents MUST have a #WeOwnVer in the header identity block.** |
| **6** | **Keeping old version on revision** | Not incrementing `r{REV}` | Forgetting to update on correction | **Every revision → increment `r{REV}` by +1. r1→r2→r3...** |
| **7** | **Assuming version from cache** | Using last session's version without checking | BP-401.5 violation | **FIRE THE TOOL. Scrape this doc. Even if you "know" the answer.** |
| **8** | **CCC-ID wrong day code** | Using `_5xxx` on Sunday | Losing track of day | **Check §8. Sunday = D7 = `_7xxx`.** |

## 10. ❓ FREQUENTLY ASKED QUESTIONS

| Question | Answer |
|:---------|:--------|
| **What version is the current workspace prompt?** | The workspace prompt follows the same #WeOwnVer as ALL governance docs. For W26: based on `v4.1.4.1` with its own `r{REV}`. Check the prompt's own header. |
| **Do I update the version when I fix a typo?** | **Yes.** Increment `r{REV}` by +1. A typo fix doesn't change the week, but it's a document revision. r1 → r2. |
| **Do I update the version when I add a new section?** | **Yes.** New content = new revision. Increment `r{REV}` by +1. |
| **What about documents created LAST week?** | They keep their week version. A doc created in W25 with `v4.1.3.1-r3` stays `v4.1.3.1` even when edited in W26. Only revision `r` increments. |
| **When does the week version change?** | **Every Monday.** @GTM 🎯 (or delegate) updates this doc with the new base version. Agents scrape this doc on Monday to get the new value. |
| **What if I'm drafting at 11:59 PM Sunday and it's W26 D7?** | Use the CURRENT week's version (W26 = `v4.1.4.1`). Even if your draft gets published Monday, the creation context is W26. The version reflects when it was drafted, not when it was pushed. |
| **Is this doc versioned too?** | **Yes.** WeOwnSeason004.md follows #WeOwnVer. Current version: `v4.1.4.1-r1`. Next week it becomes `v4.1.5.1-r1` (same doc, new week base). |
