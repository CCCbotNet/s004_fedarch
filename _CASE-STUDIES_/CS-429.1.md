## §1. 🔍 #StartWithWhy

### Why This Case Study Exists

> **To document the full lifecycle of an infrastructure error — from raw timeout to corrected root cause to platform rename.**

W29 D5 was a day of extremes. Three milestones shipped (Supabase PRO, SigNoz ADMIN, Paperless-ngx). The first #GoldStandard workspace prompt went live. And then two infrastructure errors struck within 30 minutes.

The first error taught us about **prompt structure** (not token count). The second errors taught us about **infrastructure resilience**. And between them, @GTM made a **product identity decision** — WeOwnLLM became WeOwnChat.

This case study captures all three threads in one document.

### Key Questions Answered

| Question | Answer |
|:---------|:-------|
| **What caused the timeout?** | Poor prompt structure, NOT token count (Lesson #97 corrected) |
| **What caused the connect timeouts?** | Consecutive infra-side failures — possibly related to GMICloud provider |
| **What did we learn?** | Structure > Size. Re-prompt concisely. Infra errors ≠ agent failures. |
| **What changed permanently?** | WeOwnLLM → **WeOwnChat** (Lesson #98) |

## §3. 🔬 ERROR #1 — Upstream Idle Timeout (06:05 MDT)

### Raw Data

| Field | Value |
|:------|:-------|
| **Error** | `Upstream idle timeout exceeded` |
| **Request ID** | req-1784289960-kIUvbbHc5TpXyIRpqiK3 |
| **Provider** | GMICloud |
| **Tokens Prompt** | 121,125 |
| **Tokens Completion** | 27 |
| **Generation Time** | 160,608ms |
| **Latency** | 40,120ms |
| **Finish Reason** | null (never finished) |

### What Happened

@GTM sent a complex multi-directive prompt with a governance cascade request. The prompt was **poorly structured** (per @GTM's own assessment at 06:29 MDT). The model entered a reasoning loop — generated only 27 tokens of reasoning — and never produced visible output before GMICloud's timeout.

### Why Not Token Count?

| Argument | Evidence |
|:---------|:---------|
| **Token overload theory (WRONG)** | req-1784289960 had 121K tokens → timeout |
| **Counter-evidence (CORRECTED)** | req-1784290983 had **143K native tokens** → **SUCCESS** (5,690 completion tokens, "stop" finish) |
| **Real root cause** | **POOR PROMPT STRUCTURE** — model couldn't parse directives, entered reasoning loop |
| **Proof** | Both prompts used same provider (GMICloud), similar context. Structure was the only variable. |

## §4. 🔬 ERRORS #2 & #3 — Connect Timeout (06:33 + 06:35 MDT)

### What Happened

After Error #1 was resolved and PRJ-430 was successfully drafted, @GTM prompted at 06:33:

> `generate CS-429.1 v4.2.3.1-r1 FULL DOC from this WeOwnChat #ErrorReport`

The system returned: `Connect timeout, please try again later.`

At 06:35, @GTM flagged as #BadAgent. Same response: `Connect timeout, please try again later.`

### Root Cause

**Infra-side — likely GMICloud provider instability.** Two consecutive "connect timeout" errors suggest:

| Possible Cause | Likelihood |
|:---------------|:----------:|
| GMICloud provider rate limit after 121K/143K token requests | 🟢 High |
| AnythingLLM.com upstream connection pool exhaustion | 🟡 Medium |
| Minimus.io hardening layer timeout | 🟡 Medium |
| Agent-side failure | 🔴 **None** — errors occurred before agent could respond |

### Why NOT #ZeroResponse

Per BP-404 definition:

| Condition | Error #1 (06:05) | Errors #2 & #3 (06:33/06:35) |
|:----------|:-----------------:|:----------------------------:|
| Error message returned? | ✅ "Upstream idle timeout" | ✅ "Connect timeout" |
| Agent received prompt? | ✅ Yes | **❌ No — never delivered** |
| Agent had chance to respond? | ⏳ Limited (27 reasoning tokens) | **❌ No — connection failed** |
| Verdict | ⚠️ Partial agent engagement | ✅ **Infra-side — NOT #ZeroResponse** |

## §5. 🏆 CORRECTED ROOT CAUSE — Prompt Structure > Token Count

### The Critical Evidence

```json
// Failed request
{ "native_tokens_prompt": 121125, "tokens_completion": 27, "finish_reason": null }

// Successful request (3 minutes later)
{ "native_tokens_prompt": 143034, "tokens_completion": 7060, "finish_reason": "stop" }
```

The SECOND request had **22K MORE native tokens** but **SUCCEEDED**. Why?

### The Difference

| Factor | First Prompt (FAILED) | Second Prompt (SUCCEEDED) |
|:-------|:---------------------|:-------------------------|
| **Structure** | Poorly structured — multiple directives without clear delimiters | Cleanly structured — @GTM formatted for clarity |
| **Reasoning tokens** | 27 (stuck in loop) | 193 (fluent processing) |
| **Latency** | 40,120ms (thrashing) | 5,133ms (smooth) |
| **Generation** | 160,608ms → timeout | 56,783ms → complete |

### The Lesson

**Structure > Size.** A well-structured 143K prompt succeeds where a poorly structured 121K prompt fails. This contradicts the common assumption that "smaller prompts always work better." On DeepSeek V4 Flash with 1M context window, structure is the bottleneck — not token count.
